## Что нового в 3.4.0

Этот раздел кратко перечисляет изменения версии **3.4.0** относительно 3.3.1, которые видны пользователю панели, сгруппированные по разделам руководства. Подробности по каждой функции — в соответствующем разделе ниже.

### 3. Обзор / Дашборд
- **История системных метрик: новые интервалы агрегации 12h/24h/48h** — В окне истории системных метрик к интервалам усреднения добавлены значения 12h, 24h и 48h — теперь графики (CPU, RAM, трафик, пакеты, соединения, диск, онлайн, нагрузка) можно просматривать за период до двух суток.

### 4. Inbounds: создание и общие параметры
- **Inbound: предупреждение о конфликте с зарезервированным портом Xray API** — При создании или изменении inbound панель теперь не даёт занять зарезервированный порт внутреннего Xray API (по умолчанию 62789 на 127.0.0.1): локальный TCP-inbound на этом порту на loopback отклоняется с ошибкой конфликта порта. На узлах (nodes) ограничение не действует — у них собственный Xray.
- **Tunnel/TProxy: приём streamSettings без ключа security** — Inbound типа tunnel/TProxy, у которых streamSettings не содержат блока security, теперь открываются и сохраняются без ошибки валидации.
- **Inbounds: колонка Speed (живая скорость) в списке** — В списке inbounds появилась колонка Speed с текущей скоростью трафика (отдача/загрузка) по каждому inbound.

### 5. Протоколы
- **Shadowsocks-2022: перегенерация клиентских PSK при смене метода с другим размером ключа** — Для Shadowsocks-2022: при смене метода шифрования на метод с другим размером ключа (например между aes-256 и aes-128) клиентские PSK теперь автоматически перегенерируются под новую длину при сохранении inbound. Следствие: затронутым клиентам нужно заново получить подписку (старые ссылки перестанут работать).
- **WireGuard: удалено поле workers** — Из форм WireGuard (inbound и outbound) убрано поле workers: xray-core v26.6.22 больше его не использует. Ранее сохранённые конфиги работают без изменений — поле просто игнорируется.
- **VLESS+XHTTP: флоу xtls-rprx-vision в ссылках и подписках** — Для VLESS поверх XHTTP флоу xtls-rprx-vision теперь корректно попадает в ссылки и подписки (в том числе для XHTTP+REALITY и в формате Clash/Mihomo). Раньше панель сохраняла flow, но клиенты получали конфигурацию без vision.

### 6. Транспорт (Stream Settings)
- **XHTTP: переименование sessionID-полей + Session ID Table / Length** — В транспорте XHTTP поля сессии переименованы: Session ID Placement и Session ID Key (раньше — Session Placement / Session Key). Добавлены два параметра. Session ID Table — набор символов для генерации идентификаторов сессии: можно выбрать предопределённый (ALPHABET, Base62, hex, number и др.) или ввести произвольную ASCII-строку; пусто — значение по умолчанию xray-core. Session ID Length — длина или диапазон (например 8-16) генерируемых идентификаторов; учитывается только когда задан Session ID Table, минимум должен быть больше 0. Старые сохранённые inbound мигрируются автоматически.
- **Inbound: пресет Real client IP для определения реального IP за CDN/релеем** — В настройках сокета (sockopt) inbound появился выбор Real client IP — пресет для определения реального IP посетителя, когда трафик приходит через CDN или релей (иначе фиксируется адрес посредника). Доступны три варианта: Off / direct (без обработки), Cloudflare CDN (доверять X-Forwarded-For) и L4 relay / Spectrum (PROXY) (принимать заголовок PROXY-протокола). Пресеты взаимоисключающие и предупреждают, если выбранный транспорт их не поддерживает. Эти поля никогда не отправляются клиентам в подписках.
- **gRPC: заголовок Trusted X-Forwarded-For теперь учитывается** — Заголовок Trusted X-Forwarded-For теперь учитывается и на gRPC-транспорте (раньше — только WebSocket, HTTPUpgrade и XHTTP). Для gRPC-inbound панель больше не показывает предупреждение о неподдерживаемом заголовке.

### 7. Безопасность соединения: TLS, XTLS и REALITY
- **TLS: новые поля Verify Peer Cert By Name, Curve Preferences, Master Key Log, ECH Sockopt** — Verify Peer Cert By Name — имена (через запятую), по которым клиент проверяет сертификат сервера вместо SNI; современная замена удалённого allowInsecure, передаётся в ссылках и подписках, на сервер не пишется. Curve Preferences — ограничение и порядок кривых обмена ключами TLS (например X25519MLKEM768, X25519); пусто — значения по умолчанию. Master Key Log — путь для записи TLS-ключей (формат SSLKEYLOGFILE) для отладки в Wireshark; в проде оставлять пустым. ECH Sockopt — параметры сокета для получения ECH-конфигурации (dialerProxy, Domain Strategy, TCP Fast Open, Multipath TCP).
- **REALITY: ограничение скорости fallback (Limit Fallback) и Master Key Log** — Для каждого направления (Upload и Download) задаются: After Bytes — сколько байт пропустить на полной скорости перед началом ограничения (0 — ограничивать с первого байта); Bytes Per Sec — потолок скорости fallback-трафика, чтобы зонды не использовали сервер как бесплатный канал (0 — без ограничения, отключает направление); Burst Bytes Per Sec — запас для кратковременных всплесков. Также добавлено поле Master Key Log (путь к файлу SSLKEYLOGFILE для отладки).
- **TLS: кнопки заполнения Pinned Peer Cert SHA-256 из сертификата inbound и по SNI** — Рядом с полем Pinned Peer Cert SHA-256 теперь две кнопки автозаполнения вместо прежней кнопки случайного хэша. Первая подставляет SHA-256 сертификата самого inbound. Вторая получает хэш живого сертификата сервера, выполнив TLS-подключение по указанному SNI (serverName должен быть заполнен). Полученные хэши добавляются в поле (через запятую) и попадают в ссылки для закрепления сертификата у клиента.
- **TLS: OCSP-stapling по умолчанию выключен для новых сертификатов inbound** — Для новых inbound OCSP stapling по умолчанию выключен (интервал 0). Это убирает ошибки в логах xray для сертификатов без OCSP-респондера (например Let's Encrypt). Поле осталось — его можно включить для сертификатов, которые поддерживают stapling.
- **REALITY: совместимость с полем dest (алиас target)** — Если REALITY-inbound был создан с полем dest (старыми версиями панели, через API или внешними инструментами), теперь он корректно загружается: значение dest подставляется в поле Target. Раньше Target оказывался пустым, и пересохранение ломало REALITY.

### 8. Клиенты
- **Вкладка «Links» в редакторе клиента (внешние ссылки и подписки)** — На ней кнопкой **Add External Link** добавляют сторонние share-ссылки (`vless://`, `vmess://`, `trojan://`, `ss://`, `hysteria2://`, `wireguard://`), а кнопкой **Add External Subscription** — URL удалённых подписок. Всё перечисленное подмешивается в выдачу подписки данного клиента (форматы raw, JSON и Clash): ссылки добавляются как есть, а удалённые подписки панель периодически скачивает и объединяет их конфигурации с собственными.
- **Поле «Лимит IP» теперь отключается без Fail2ban** — Поле **Лимит IP** теперь работает только при установленном и активном Fail2ban. Если Fail2ban не установлен (или система Windows, или функция отключена на сервере), поле редактора клиента блокируется, а при наведении показывается подсказка с предложением установить Fail2ban из bash-меню `x-ui`. При обновлении панели у клиентов на серверах без Fail2ban сохранённый лимит IP обнуляется, так как он там всё равно не применялся.
- **Удаление непривязанных клиентов, экспорт и импорт клиентов** — В меню **Ещё** на странице **Клиенты** добавлены три операции. **Экспорт клиентов** показывает JSON-список всех клиентов (в формате `{client, inboundIds}`) с кнопками копирования и скачивания (`clients-export.json`). **Импорт клиентов** принимает такой же JSON: клиенты с привязками воссоздаются и привязываются к inbound, клиенты без привязок восстанавливаются как отдельные записи, а уже существующие email не перезаписываются (они отмечаются как пропущенные). **Удалить непривязанных клиентов** удаляет всех клиентов, не привязанных ни к одному inbound, вместе с их трафиком, журналом IP и внешними ссылками; действие необратимо.
- **Журнал IP клиента: время подключения и имя ноды** — В журнале IP клиента (кнопка просмотра рядом с полем «Лимит IP» и в карточке «Информация о клиенте») каждая запись теперь содержит, помимо самого IP, время последнего обращения и метку ноды (`@ имя_ноды`), через которую зафиксировано подключение, — в мультипанельной конфигурации видно, через какой узел подключался клиент.
- **Сброс метки группы у клиента в одиночном редакторе** — Теперь если в редакторе одного клиента очистить поле **Группа** и сохранить, метка группы корректно снимается — ранее клиент мог продолжать отображаться под прежней группой до повторного сохранения.
- **Список клиентов обновляется автоматически (фоновый опрос)** — Список клиентов теперь обновляется автоматически: панель раз в несколько секунд подтягивает актуальную страницу, поэтому вновь подключившиеся клиенты и изменившийся порядок сортировки появляются без ручного обновления.

### 10. Подписки (Subscription)
- **Управляемые Hosts: переопределение subscription-ссылок по хостам** — В версии 3.4.0 добавлен раздел Hosts (пункт бокового меню). Для каждого inbound можно задать один или несколько Host-эндпоинтов, которые подставляются в subscription-ссылки вместо адреса, порта и TLS-параметров самого inbound — это удобно для раздачи трафика через CDN или релей. У хоста задаются Remark и описание, привязка к inbound, Address (пусто — наследуется адрес inbound) и Port (0 — наследуется порт inbound), параметры Security (same/tls/none/reality), а также Host header, Path, Mux, Sockopt, Final Mask, исключение из форматов подписки (raw/json/clash) и параметры для Clash/mihomo. Хосты упорядочиваются внутри inbound и поддерживают массовые операции.
- **Remark Template заменил конструктор модели ремарки; переменные {{VAR}}** — Прежний конструктор имени профиля (выбор Inbound/Email/External Proxy и разделителя) заменён полем «Remark Template». В нём вы пишете произвольный формат имени, вставляя переменные кнопкой: идентификация клиента ({{EMAIL}}, {{INBOUND}}, {{HOST}}, {{ID}}, {{SUB_ID}}, {{COMMENT}}, {{TELEGRAM_ID}}), трафик ({{TRAFFIC_USED}}, {{TRAFFIC_LEFT}}, {{TRAFFIC_TOTAL}}, {{UP}}, {{DOWN}}, {{USAGE_PERCENTAGE}}) и срок/статус ({{DAYS_LEFT}}, {{TIME_LEFT}}, {{EXPIRE_DATE}}, {{JALALI_EXPIRE_DATE}}, {{STATUS}}, {{STATUS_EMOJI}}). Переменные подставляются индивидуально для каждого клиента при генерации подписки, доступен предпросмотр. Сегменты, разделённые символом «|», с безлимитными значениями скрываются автоматически, а сведения о расходе и сроке показываются только на первой ссылке клиента. Если поле оставить пустым, используется прежняя модель ремарки.
- **Per-client внешние ссылки и удалённые подписки (вкладка Links)** — Здесь для отдельного клиента можно прикрепить сторонние share-ссылки (Add External Link) и адреса внешних подписок (Add External Subscription) — они будут включены в его собственную подписку (форматы RAW, JSON и Clash). Внешние подписки скачиваются панелью и объединяются с конфигурациями клиента. Это удобно, чтобы выдать клиенту дополнительные серверы поверх ваших inbound.
- **Happ: скрытие настроек сервера в клиенте (Hide server settings)** — На вкладке Happ настроек подписки добавлен переключатель «Hide server settings». Когда он включён, в клиенте Happ скрывается возможность просматривать и изменять параметры сервера. Опция действует только для клиента Happ.
- **Имя узла больше не дописывается к имени профиля в подписке** — Имя узла (Node) больше не добавляется к имени профиля в подписке. В клиентском приложении показывается только ремарка inbound, заданная администратором, без внутреннего суффикса вида «@имя-узла».
- **Метка модели ремарки переименована Other → External Proxy (затем заменена шаблоном)** — Отдельно документировать не нужно: переименование пункта модели ремарки «Other» в «External Proxy» поглощено переходом на новое поле Remark Template, где конструктор модели из UI убран.
- **Корректность ссылок подписки: SS2022, Shadowrocket, SIP002 obfs, XHTTP в Clash** — Улучшена совместимость генерируемых ссылок подписки: исправлено кодирование SS2022, deep-link для Shadowrocket, выдача Shadowsocks+obfs в формате SIP002 (плагин obfs-local) и полный набор полей XHTTP в подписках Clash/Mihomo. Отдельных настроек не требуется — ссылки просто корректнее распознаются клиентами.
- **Модель ремарок подписки: пункт «Other» переименован в «External Proxy»** — В настройках подписки в модели ремарок пункт **«Other»** переименован в **«External Proxy»** (источник — ремарка внешнего прокси). Поведение не изменилось, существующие настройки остаются совместимыми.
- **Подписки: выбор переменных ремарки кликом по чипам (Remark variable picker)** — Рядом с полем Remark Template доступен сгруппированный набор переменных-чипов: клик по переменной {{VAR}} вставляет её в шаблон, при наведении показывается описание. В полях ремарки и имени хоста допускается и упрощённая запись в одинарных скобках — {DATA_LEFT}, {EXPIRE_DATE}, {PROTOCOL}, {TRANSPORT} и т. п.; панель автоматически преобразует её во внутренний формат {{...}}.

### 11. Xray: маршрутизация, outbounds, DNS и расширения
- **Маршрутизация и Outbounds вынесены в отдельные пункты бокового меню** — Начиная с этой версии **«Исходящие» (Outbounds)** и **«Маршрутизация» (Routing)** вынесены в отдельные пункты бокового меню (сразу под «Хосты»), у каждого свой адрес — `/outbound` и `/routing`. Раньше маршрутизация открывалась внутри подменю «Конфигурации Xray», а outbounds — как вкладка страницы Xray. В подменю «Конфигурации Xray» теперь остаются только: Основное, Балансировщик, DNS и Расширенный шаблон. Прямые ссылки на `/outbound` и `/routing` и перезагрузка страницы работают как ожидается.
- **Правила маршрутизации можно включать и отключать переключателем** — Отдельное правило маршрутизации теперь можно временно **отключить** переключателем, не удаляя его. В таблице правил есть столбец **«Включить»** с переключателем, в форме правила поле «Включить» — тоже переключатель. Отключённое правило не попадает в итоговую конфигурацию Xray. Служебное правило статистики (`api`) отключить нельзя — его переключатель заблокирован.
- **Экспорт правил маршрутизации и outbounds открывается в модальном окне предпросмотра** — Кнопка **«Экспорт»** правил маршрутизации и исходящих теперь не скачивает файл сразу, а открывает модальное окно с предпросмотром JSON и кнопками **«Копировать»** и **«Скачать»**. На вкладке «Маршрутизация» «Импорт» и «Экспорт» собраны в выпадающее меню **«ещё»** (как на вкладке Outbounds).
- **Тест всех исходящих теперь проверяет и outbounds из подписок; direct/dns больше не тестируются** — Кнопка **«Тест всех»** на странице «Исходящие» теперь проверяет и outbounds, полученные из подписок (таблица «из подписок») — их строки также подсвечиваются результатом. При этом outbounds `freedom` («direct») и `dns` больше не тестируются ни в одном режиме (это не прокси): кнопка теста у них недоступна, а «Тест всех» их пропускает.
- **FinalMask: пофрагментные массивы Lengths/Delays вместо одиночных Length/Delay** — В маске fragment (FinalMask) одиночные поля Length и Delay заменены списками Lengths и Delays: для каждого сегмента можно задать отдельный диапазон длины (например 100-200) и задержки в мс (например 10-20 или 0). Строки можно добавлять и удалять; ранее сохранённые значения переносятся автоматически.
- **Loopback outbound: добавлен блок Sniffing** — У outbound типа loopback появился блок Sniffing с теми же параметрами, что и у inbound: включение, destOverride, Metadata Only, Route Only и список исключённых доменов.
- **Hysteria2 / Salamander: режим Gecko (packetSize) и TLS для маски Realm** — В маске UDP (FinalMask) для Hysteria2 расширены возможности. У маски Salamander добавлен селектор Mode: режим Gecko добавляет случайное дополнение пакетов с полями Min/Max размера (от 1 до 2048, по умолчанию 512-1200) для защиты от анализа длины пакетов. У маски Realm появился необязательный блок TLS Config: Server Name (SNI), ALPN (h3/h2/http/1.1), Fingerprint и Allow Insecure.
- **Импорт share-ссылки в outbound сохраняет настройки xmux** — При импорте outbound из share-ссылки теперь сохраняются настройки мультиплексора **xmux** (XHTTP): раньше они молча терялись. После импорта значения подставляются в подформу XMUX.
- **Теги outbounds из подписок сохраняют не-ASCII символы (кириллица)** — Теги outbounds, полученных из подписок, теперь сохраняют не-ASCII символы (например, кириллицу) и остаются читаемыми, а не сводятся к одним цифрам.

### 12. Узлы (мультипанель, master/slave)
- **Узлы: новый режим проверки TLS — Mutual TLS (клиентский сертификат)** — В форме узла режим проверки TLS теперь имеет четыре варианта: Verify (системный CA), Pin (фиксация сертификата по SHA-256), Skip (без проверки) и новый Mutual TLS (клиентский сертификат). В режиме Mutual TLS панель дополнительно подтверждает себя узлу клиентским сертификатом, выпущенным её собственным CA; API-токен для такого узла становится необязательным. Чтобы включить mTLS: на узле задайте режим Mutual TLS, скопируйте CA управляющей панели из раздела Node mTLS, пропишите его на узле как доверенный родительский CA и перезапустите узел.
- **Узлы: раздел «Node mTLS» — копирование CA панели и доверенный родительский CA** — На странице Узлы добавлен раздел Node mTLS для настройки взаимной TLS-аутентификации между панелями. Кнопка «Скопировать CA этой панели» копирует корневой сертификат панели в буфер обмена — его нужно передать управляемым узлам, которые будут проверять клиентский сертификат панели. Поле «Доверенный родительский CA» используется, когда сама панель является узлом: вставьте сюда CA управляющей панели, чтобы требовать её клиентский сертификат, и перезапустите панель. Взаимный TLS включается по желанию; если поля пусты, узлы работают по прежней схеме — только с API-токеном.
- **Маршрутизация исходящего подключения панели к узлу (Connection outbound)** — В форме узла появилось поле **«Connection outbound»** (исходящее подключение). Если выбрать в нём тег Xray-outbound, трафик панели к API этого узла пойдёт через указанный outbound (панель сама добавит в рабочую конфигурацию мост-inbound на loopback и применит его без перезапуска). Пустое значение = прямое подключение. В списке теги сгруппированы на «Исходящие» и «Балансировщики», blackhole-исходящие скрыты.
- **Узлы: маршрутизация трафика панель→узел через выбранный outbound («Connection outbound»)** — В форме узла появилось поле «Connection outbound»: оно позволяет направить трафик обращений панели к узлу через выбранный Xray outbound (доступны обычные outbounds и балансировщики). Панель автоматически добавляет в рабочую конфигурацию loopback-bridge inbound и применяет изменения без перезапуска. Оставьте поле пустым для прямого подключения.
- **Узлы: удаление узла блокируется, пока к нему привязаны inbounds** — Удалить узел можно только после того, как с него сняты все inbounds. Если к узлу всё ещё привязан хотя бы один inbound, панель отклонит удаление с ошибкой — сначала открепите или удалите эти inbounds, затем удалите узел.
- **Узлы: на странице узлов показывается живая скорость inbound'ов, размещённых на узле** — На странице Узлы для inbound'ов, размещённых на узле, теперь показываются онлайн-клиенты, счётчики и текущая скорость передачи. Чип «завершённые» (ended) считает только истёкших и исчерпавших трафик клиентов (отключённые в него больше не входят).

### 14. Telegram-бот
- **Уведомления: новая шина событий с каналами Telegram и Email (SMTP)** — Добавлена система событийных уведомлений с двумя каналами доставки — Telegram и Email. На вкладке уведомлений события сгруппированы карточками: Outbound (падение/восстановление), Xray Core (аварийное завершение), Nodes (узел недоступен/восстановлен), System (высокая загрузка CPU и памяти с настраиваемым порогом в %), Security (попытка входа). У каждой группы есть мастер-переключатель и счётчик выбранных событий. Набор включённых событий настраивается раздельно для Telegram и Email; по умолчанию включены «попытка входа» и «высокая загрузка CPU».
- **Уведомления: новый канал Email/SMTP и настройки SMTP-сервера** — Добавлен новый канал уведомлений по электронной почте через SMTP. На вкладке настроек SMTP задаются: включение email-уведомлений, хост и порт SMTP (по умолчанию 587), имя пользователя, пароль (хранится скрыто), список получателей (через запятую) и тип шифрования — none, STARTTLS (по умолчанию) или TLS. Кнопка «Отправить тестовое письмо» проверяет соединение и показывает, на каком этапе (подключение, аутентификация, отправка) возникла ошибка. На второй вкладке выбираются события, о которых будут приходить письма.
- **Уведомления: оповещение о высокой загрузке памяти (порог RAM)** — К оповещениям о высокой загрузке CPU добавлено оповещение о высокой загрузке оперативной памяти. В группе событий «System» появилось «Memory high (%)» с собственным полем порога (по умолчанию 80%); панель раз в минуту проверяет загрузку RAM и при превышении порога отправляет уведомление в выбранные каналы.

### 15. Гео-базы (geoip / geosite и кастомные)
- **Обновление гео-баз: статус по каждому файлу и пропуск перезапуска без изменений** — Обновление гео-баз (geoip/geosite, включая наборы IR и RU) теперь показывает статус по каждому файлу: обновлён, уже актуален или ошибка загрузки. Перезапуск Xray (а значит и разрыв активных соединений) происходит только если хотя бы один файл действительно был обновлён; при отсутствии изменений панель не перезапускается. То же поведение у команды x-ui update-all-geofiles.

### 16. Эксплуатация: бэкапы, логи, обновление, CLI
- **IP-лимит клиента работает только при установленном fail2ban; поле блокируется иначе** — Ограничение по количеству IP у клиента теперь действует только если на сервере установлен fail2ban. Если его нет, поле «IP Limit» в форме клиента и при массовом добавлении становится недоступным с поясняющей подсказкой (на Windows — отдельным сообщением), а ранее заданные лимиты на таких серверах автоматически обнуляются, поскольку они всё равно не применялись. Блокировка fail2ban теперь распространяется и на TCP, и на UDP.
- **Автоматическая установка fail2ban при установке и обновлении панели** — При установке и обновлении панели на обычном сервере fail2ban теперь устанавливается и настраивается автоматически (раньше это происходило только в Docker), чтобы функция IP-лимита работала из коробки. Поведением управляет переменная окружения XUI_ENABLE_FAIL2BAN: настройка выполняется, если переменная не задана или равна true. Ручной запуск доступен командой x-ui setup-fail2ban; ошибка fail2ban не прерывает установку или обновление.
- **Переопределение порта панели через переменную XUI_PORT** — Добавлена переменная окружения XUI_PORT, которая задаёт порт веб-панели только на время работы текущего процесса, не изменяя сохранённое значение webPort в базе. Допустимы значения от 1 до 65535; пустое, некорректное или выходящее за диапазон значение игнорируется (используется webPort) с предупреждением в логе. При использовании Docker с bridge-сетью публикуемый порт контейнера должен совпадать с XUI_PORT, например XUI_PORT=8080 и ports: «8080:8080».
- **CLI: флаги -webCert/-webCertKey теперь применяются в подкоманде setting** — В CLI флаги -webCert и -webCertKey теперь работают и в подкоманде x-ui setting (раньше они молча игнорировались и панель оставалась без HTTPS). Указав их, можно сразу задать путь к сертификату и ключу веб-панели, не вызывая отдельную подкоманду cert.
- **Имя файла резервной копии БД формируется по адресу сервера** — Файлы резервной копии базы данных теперь называются по адресу сервера, а не фиксированным x-ui.db / x-ui.dump. При скачивании из браузера имя берётся из адреса панели в адресной строке, иначе — из настроенного веб-домена, а при его отсутствии — из публичного IP (сначала IPv4, затем IPv6). Так бэкапы с разных серверов легко различать. Расширение остаётся .db для SQLite и .dump для PostgreSQL.
- **Поддержка установки и обновления на хостах только с IPv6** — Скрипты установки и обновления теперь корректно работают на серверах только с IPv6: загрузка релиза и вспомогательных файлов больше не принудительно использует IPv4, поэтому панель можно установить и обновить на хосте без IPv4-адреса.
