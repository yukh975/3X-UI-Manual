## تازه‌ها در 3.4.0

این بخش به‌اختصار تغییرات نسخه **3.4.0** نسبت به 3.3.1 را که برای کاربر پنل قابل مشاهده‌اند، گروه‌بندی‌شده بر اساس بخش‌های راهنما فهرست می‌کند. جزئیات هر قابلیت در بخش مربوطه در پایین آمده است.

### 3. نمای کلی / داشبورد
- **تاریخچه متریک‌های سیستم: بازه‌های تجمیع جدید 12h/24h/48h** — در پنجره تاریخچه متریک‌های سیستم، به بازه‌های میانگین‌گیری مقادیر 12h، 24h و 48h افزوده شد — اکنون نمودارها (CPU، RAM، ترافیک، بسته‌ها، اتصالات، دیسک، آنلاین، بار) را می‌توان برای دوره‌ای تا دو شبانه‌روز مشاهده کرد.

### 4. Inbounds: ایجاد و پارامترهای عمومی
- **Inbound: هشدار تداخل با پورت رزروشده Xray API** — هنگام ایجاد یا تغییر inbound، پنل اکنون اجازه نمی‌دهد پورت رزروشده Xray API داخلی اشغال شود (به‌صورت پیش‌فرض 62789 روی 127.0.0.1): یک inbound محلی TCP روی این پورت در loopback با خطای تداخل پورت رد می‌شود. روی نودها (nodes) این محدودیت اعمال نمی‌شود — آن‌ها Xray مخصوص خود را دارند.
- **Tunnel/TProxy: پذیرش streamSettings بدون کلید security** — inbound از نوع tunnel/TProxy که streamSettings آن‌ها بلوک security ندارد، اکنون بدون خطای اعتبارسنجی باز و ذخیره می‌شوند.
- **Inbounds: ستون Speed (سرعت زنده) در فهرست** — در فهرست inbounds ستون Speed با سرعت جاری ترافیک (آپلود/دانلود) برای هر inbound اضافه شد.

### 5. پروتکل‌ها
- **Shadowsocks-2022: تولید مجدد PSK کلاینت‌ها هنگام تغییر method به method با اندازه کلید متفاوت** — برای Shadowsocks-2022: هنگام تغییر method رمزنگاری به method با اندازه کلید متفاوت (مثلاً بین aes-256 و aes-128)، PSK کلاینت‌ها اکنون به‌صورت خودکار هنگام ذخیره inbound برای طول جدید بازتولید می‌شوند. پیامد: کلاینت‌های تحت‌تأثیر باید اشتراک را دوباره دریافت کنند (لینک‌های قدیمی از کار می‌افتند).
- **WireGuard: فیلد workers حذف شد** — از فرم‌های WireGuard (inbound و outbound) فیلد workers برداشته شد: xray-core v26.6.22 دیگر از آن استفاده نمی‌کند. پیکربندی‌های قبلاً ذخیره‌شده بدون تغییر کار می‌کنند — فیلد به‌سادگی نادیده گرفته می‌شود.
- **VLESS+XHTTP: flow xtls-rprx-vision در لینک‌ها و اشتراک‌ها** — برای VLESS روی XHTTP، flow با مقدار xtls-rprx-vision اکنون به‌درستی در لینک‌ها و اشتراک‌ها قرار می‌گیرد (از جمله برای XHTTP+REALITY و در قالب Clash/Mihomo). پیش‌تر پنل flow را ذخیره می‌کرد اما کلاینت‌ها پیکربندی را بدون vision دریافت می‌کردند.

### 6. transport (Stream Settings)
- **XHTTP: تغییر نام فیلدهای sessionID + Session ID Table / Length** — در transport XHTTP فیلدهای نشست تغییر نام یافتند: Session ID Placement و Session ID Key (پیش‌تر — Session Placement / Session Key). دو پارامتر اضافه شد. Session ID Table — مجموعه‌ای از کاراکترها برای تولید شناسه‌های نشست: می‌توان یک مقدار از پیش تعریف‌شده (ALPHABET، Base62، hex، number و غیره) انتخاب کرد یا یک رشته ASCII دلخواه وارد نمود؛ خالی — مقدار پیش‌فرض xray-core. Session ID Length — طول یا بازه (مثلاً 8-16) شناسه‌های تولیدشده؛ تنها زمانی لحاظ می‌شود که Session ID Table تعیین شده باشد، حداقل باید بزرگ‌تر از 0 باشد. inbound‌های قدیمی ذخیره‌شده به‌صورت خودکار مهاجرت می‌کنند.
- **Inbound: پیش‌تنظیم Real client IP برای تشخیص IP واقعی پشت CDN/رله** — در تنظیمات سوکت (sockopt) inbound گزینه Real client IP اضافه شد — یک پیش‌تنظیم برای تشخیص IP واقعی بازدیدکننده، زمانی که ترافیک از طریق CDN یا رله می‌رسد (در غیر این صورت آدرس واسطه ثبت می‌شود). سه گزینه در دسترس است: Off / direct (بدون پردازش)، Cloudflare CDN (اعتماد به X-Forwarded-For) و L4 relay / Spectrum (PROXY) (پذیرش هدر پروتکل PROXY). پیش‌تنظیم‌ها متقابلاً انحصاری‌اند و اگر transport انتخاب‌شده از آن‌ها پشتیبانی نکند هشدار می‌دهند. این فیلدها هرگز در اشتراک‌ها برای کلاینت‌ها ارسال نمی‌شوند.
- **gRPC: هدر Trusted X-Forwarded-For اکنون لحاظ می‌شود** — هدر Trusted X-Forwarded-For اکنون روی transport gRPC نیز لحاظ می‌شود (پیش‌تر — فقط WebSocket، HTTPUpgrade و XHTTP). برای inbound gRPC پنل دیگر هشدار هدر پشتیبانی‌نشده را نمایش نمی‌دهد.

### 7. امنیت اتصال: TLS، XTLS و REALITY
- **TLS: فیلدهای جدید Verify Peer Cert By Name، Curve Preferences، Master Key Log، ECH Sockopt** — Verify Peer Cert By Name — نام‌هایی (جداشده با ویرگول) که کلاینت گواهی سرور را به‌جای SNI با آن‌ها بررسی می‌کند؛ جایگزین مدرن allowInsecure حذف‌شده، در لینک‌ها و اشتراک‌ها منتقل می‌شود و روی سرور نوشته نمی‌شود. Curve Preferences — محدودسازی و ترتیب منحنی‌های تبادل کلید TLS (مثلاً X25519MLKEM768، X25519)؛ خالی — مقادیر پیش‌فرض. Master Key Log — مسیری برای نوشتن کلیدهای TLS (قالب SSLKEYLOGFILE) برای اشکال‌زدایی در Wireshark؛ در محیط تولید خالی نگه دارید. ECH Sockopt — پارامترهای سوکت برای دریافت پیکربندی ECH (dialerProxy، Domain Strategy، TCP Fast Open، Multipath TCP).
- **REALITY: محدودسازی سرعت fallback (Limit Fallback) و Master Key Log** — برای هر جهت (Upload و Download) تعیین می‌شود: After Bytes — چند بایت با سرعت کامل عبور داده شود پیش از آغاز محدودسازی (0 — محدودسازی از اولین بایت)؛ Bytes Per Sec — سقف سرعت ترافیک fallback، تا پروب‌ها از سرور به‌عنوان کانال رایگان استفاده نکنند (0 — بدون محدودیت، جهت را غیرفعال می‌کند)؛ Burst Bytes Per Sec — ذخیره برای جهش‌های کوتاه‌مدت. همچنین فیلد Master Key Log اضافه شد (مسیر فایل SSLKEYLOGFILE برای اشکال‌زدایی).
- **TLS: دکمه‌های پرکردن Pinned Peer Cert SHA-256 از گواهی inbound و بر اساس SNI** — کنار فیلد Pinned Peer Cert SHA-256 اکنون به‌جای دکمه قبلی هش تصادفی، دو دکمه پرکردن خودکار وجود دارد. اولی SHA-256 گواهی خودِ inbound را جای می‌گذارد. دومی هش گواهی زنده سرور را با برقراری اتصال TLS بر اساس SNI تعیین‌شده دریافت می‌کند (serverName باید پر شده باشد). هش‌های به‌دست‌آمده به فیلد افزوده می‌شوند (جداشده با ویرگول) و در لینک‌ها برای pin کردن گواهی نزد کلاینت قرار می‌گیرند.
- **TLS: OCSP-stapling به‌صورت پیش‌فرض برای گواهی‌های جدید inbound غیرفعال است** — برای inbound‌های جدید، OCSP stapling به‌صورت پیش‌فرض غیرفعال است (بازه 0). این کار خطاها را در لاگ‌های xray برای گواهی‌های بدون OCSP-responder (مثلاً Let's Encrypt) حذف می‌کند. فیلد باقی مانده است — می‌توان آن را برای گواهی‌هایی که از stapling پشتیبانی می‌کنند فعال کرد.
- **REALITY: سازگاری با فیلد dest (نام مستعار target)** — اگر یک inbound از نوع REALITY با فیلد dest ساخته شده باشد (با نسخه‌های قدیمی پنل، از طریق API یا ابزارهای خارجی)، اکنون به‌درستی بارگذاری می‌شود: مقدار dest در فیلد Target جای‌گذاری می‌شود. پیش‌تر Target خالی می‌ماند و ذخیره مجدد، REALITY را خراب می‌کرد.

### 8. کلاینت‌ها
- **زبانه «Links» در ویرایشگر کلاینت (لینک‌ها و اشتراک‌های خارجی)** — در آن با دکمه **Add External Link** لینک‌های share شخص ثالث (`vless://`، `vmess://`، `trojan://`، `ss://`، `hysteria2://`، `wireguard://`) و با دکمه **Add External Subscription** آدرس‌های اشتراک‌های راه دور افزوده می‌شوند. همه موارد فهرست‌شده در خروجی اشتراک همان کلاینت ادغام می‌شوند (قالب‌های raw، JSON و Clash): لینک‌ها همان‌گونه که هستند افزوده می‌شوند و اشتراک‌های راه دور را پنل به‌صورت دوره‌ای دانلود کرده و پیکربندی‌های آن‌ها را با پیکربندی‌های خود تلفیق می‌کند.
- **فیلد «Лимит IP» اکنون بدون Fail2ban غیرفعال می‌شود** — فیلد **Лимит IP** اکنون تنها در صورت نصب و فعال بودن Fail2ban کار می‌کند. اگر Fail2ban نصب نباشد (یا سیستم Windows باشد، یا این قابلیت روی سرور غیرفعال شده باشد)، فیلد ویرایشگر کلاینت قفل می‌شود و هنگام قرار دادن نشانگر، راهنمایی با پیشنهاد نصب Fail2ban از منوی bash دستور `x-ui` نمایش داده می‌شود. هنگام به‌روزرسانی پنل، برای کلاینت‌های روی سرورهای بدون Fail2ban، محدودیت IP ذخیره‌شده صفر می‌شود، چون در آنجا به‌هرحال اعمال نمی‌شد.
- **حذف کلاینت‌های بدون پیوند، صادرات و واردات کلاینت‌ها** — در منوی **بیشتر** در صفحه **کلاینت‌ها** سه عملیات افزوده شد. **صادرات کلاینت‌ها** یک فهرست JSON از همه کلاینت‌ها (در قالب `{client, inboundIds}`) با دکمه‌های کپی و دانلود (`clients-export.json`) نمایش می‌دهد. **واردات کلاینت‌ها** همان JSON را می‌پذیرد: کلاینت‌های دارای پیوند بازآفرینی شده و به inbound پیوند می‌خورند، کلاینت‌های بدون پیوند به‌عنوان رکوردهای جداگانه بازیابی می‌شوند و email‌های از پیش موجود بازنویسی نمی‌شوند (آن‌ها به‌عنوان رد شده علامت می‌خورند). **حذف کلاینت‌های بدون پیوند** همه کلاینت‌هایی را که به هیچ inbound پیوند نخورده‌اند، همراه با ترافیک، journal IP و لینک‌های خارجی آن‌ها حذف می‌کند؛ این عمل برگشت‌ناپذیر است.
- **journal IP کلاینت: زمان اتصال و نام نود** — در journal IP کلاینت (دکمه مشاهده کنار فیلد «Лимит IP» و در کارت «اطلاعات کلاینت») هر رکورد اکنون علاوه بر خود IP، شامل زمان آخرین مراجعه و برچسب نود (`@ имя_ноды`) است که از طریق آن اتصال ثبت شده — در پیکربندی مولتی‌پنل مشخص می‌شود کلاینت از طریق کدام نود متصل شده است.
- **بازنشانی برچسب گروه کلاینت در ویرایشگر تکی** — اکنون اگر در ویرایشگر یک کلاینت فیلد **گروه** پاک شود و ذخیره گردد، برچسب گروه به‌درستی برداشته می‌شود — پیش‌تر کلاینت ممکن بود تا ذخیره مجدد، همچنان زیر گروه قبلی نمایش داده شود.
- **فهرست کلاینت‌ها به‌صورت خودکار به‌روز می‌شود (پایش پس‌زمینه)** — فهرست کلاینت‌ها اکنون به‌صورت خودکار به‌روز می‌شود: پنل هر چند ثانیه یک‌بار صفحه فعلی را دریافت می‌کند، بنابراین کلاینت‌های تازه‌متصل‌شده و ترتیب مرتب‌سازی تغییریافته بدون به‌روزرسانی دستی ظاهر می‌شوند.

### 10. اشتراک‌ها (Subscription)
- **Managed Hosts: بازنویسی لینک‌های اشتراک بر اساس میزبان‌ها** — در نسخه 3.4.0 بخش Hosts افزوده شد (یک گزینه از منوی کناری). برای هر inbound می‌توان یک یا چند نقطه پایانی Host تعریف کرد که به‌جای آدرس، پورت و پارامترهای TLS خودِ inbound در لینک‌های اشتراک قرار می‌گیرند — این برای توزیع ترافیک از طریق CDN یا رله مناسب است. برای میزبان، Remark و توضیحات، پیوند به inbound، Address (خالی — آدرس inbound به ارث می‌رسد) و Port (0 — پورت inbound به ارث می‌رسد)، پارامترهای Security (same/tls/none/reality) و همچنین Host header، Path، Mux، Sockopt، Final Mask، استثنا از قالب‌های اشتراک (raw/json/clash) و پارامترهای Clash/mihomo تعیین می‌شوند. میزبان‌ها داخل inbound مرتب می‌شوند و از عملیات گروهی پشتیبانی می‌کنند.
- **Remark Template جایگزین سازنده مدل ریمارک شد؛ متغیرهای {{VAR}}** — سازنده پیشین نام پروفایل (انتخاب Inbound/Email/External Proxy و جداکننده) با فیلد «Remark Template» جایگزین شد. در آن قالب دلخواه نام را می‌نویسید و متغیرها را با دکمه درج می‌کنید: شناسایی کلاینت ({{EMAIL}}، {{INBOUND}}، {{HOST}}، {{ID}}، {{SUB_ID}}، {{COMMENT}}، {{TELEGRAM_ID}})، ترافیک ({{TRAFFIC_USED}}، {{TRAFFIC_LEFT}}، {{TRAFFIC_TOTAL}}، {{UP}}، {{DOWN}}، {{USAGE_PERCENTAGE}}) و مهلت/وضعیت ({{DAYS_LEFT}}، {{TIME_LEFT}}، {{EXPIRE_DATE}}، {{JALALI_EXPIRE_DATE}}، {{STATUS}}، {{STATUS_EMOJI}}). متغیرها برای هر کلاینت به‌صورت جداگانه هنگام تولید اشتراک جای‌گذاری می‌شوند و پیش‌نمایش در دسترس است. سگمنت‌های جداشده با نماد «|» که مقادیر بی‌نهایت دارند به‌صورت خودکار پنهان می‌شوند و اطلاعات مصرف و مهلت تنها روی اولین لینک کلاینت نمایش داده می‌شوند. اگر فیلد را خالی بگذارید، مدل ریمارک پیشین استفاده می‌شود.
- **لینک‌های خارجی و اشتراک‌های راه دور به‌ازای هر کلاینت (زبانه Links)** — اینجا برای یک کلاینت مشخص می‌توان لینک‌های share شخص ثالث (Add External Link) و آدرس‌های اشتراک‌های خارجی (Add External Subscription) را پیوست کرد — آن‌ها در اشتراک خودِ او گنجانده می‌شوند (قالب‌های RAW، JSON و Clash). اشتراک‌های خارجی توسط پنل دانلود شده و با پیکربندی‌های کلاینت تلفیق می‌شوند. این برای ارائه سرورهای اضافی به کلاینت علاوه بر inbound‌های شما مناسب است.
- **Happ: پنهان‌سازی تنظیمات سرور در کلاینت (Hide server settings)** — در زبانه Happ تنظیمات اشتراک، کلید «Hide server settings» افزوده شد. وقتی فعال باشد، در کلاینت Happ امکان مشاهده و تغییر پارامترهای سرور پنهان می‌شود. این گزینه تنها برای کلاینت Happ عمل می‌کند.
- **نام نود دیگر به نام پروفایل در اشتراک افزوده نمی‌شود** — نام نود (Node) دیگر به نام پروفایل در اشتراک افزوده نمی‌شود. در برنامه کلاینت تنها ریمارک inbound که توسط مدیر تعیین شده، بدون پسوند داخلی به شکل «@имя-узла» نمایش داده می‌شود.
- **برچسب مدل ریمارک از Other → External Proxy تغییر نام یافت (سپس با قالب جایگزین شد)** — لازم نیست جداگانه مستند شود: تغییر نام گزینه مدل ریمارک «Other» به «External Proxy» در گذار به فیلد جدید Remark Template جذب شد، جایی که سازنده مدل از UI حذف شده است.
- **درستی لینک‌های اشتراک: SS2022، Shadowrocket، SIP002 obfs، XHTTP در Clash** — سازگاری لینک‌های اشتراک تولیدشده بهبود یافت: کدگذاری SS2022، deep-link برای Shadowrocket، خروجی Shadowsocks+obfs در قالب SIP002 (افزونه obfs-local) و مجموعه کامل فیلدهای XHTTP در اشتراک‌های Clash/Mihomo اصلاح شد. تنظیمات جداگانه‌ای لازم نیست — لینک‌ها صرفاً درست‌تر توسط کلاینت‌ها شناسایی می‌شوند.
- **مدل ریمارک‌های اشتراک: گزینه «Other» به «External Proxy» تغییر نام یافت** — در تنظیمات اشتراک در مدل ریمارک‌ها، گزینه **«Other»** به **«External Proxy»** تغییر نام یافت (منبع — ریمارک پراکسی خارجی). رفتار تغییر نکرده است و تنظیمات موجود سازگار باقی می‌مانند.
- **اشتراک‌ها: انتخاب متغیرهای ریمارک با کلیک روی چیپ‌ها (Remark variable picker)** — کنار فیلد Remark Template مجموعه‌ای گروه‌بندی‌شده از متغیرها به‌شکل چیپ در دسترس است: کلیک روی متغیر {{VAR}} آن را در قالب درج می‌کند و هنگام قرار دادن نشانگر، توضیحات نمایش داده می‌شود. در فیلدهای ریمارک و نام میزبان، نگارش ساده‌شده با کروشه تکی نیز مجاز است — {DATA_LEFT}، {EXPIRE_DATE}، {PROTOCOL}، {TRANSPORT} و غیره؛ پنل آن را به‌صورت خودکار به قالب داخلی {{...}} تبدیل می‌کند.

### 11. Xray: مسیریابی، outbounds، DNS و افزونه‌ها
- **مسیریابی و Outbounds به گزینه‌های جداگانه منوی کناری منتقل شدند** — از این نسخه به بعد **«خروجی‌ها» (Outbounds)** و **«مسیریابی» (Routing)** به گزینه‌های جداگانه منوی کناری منتقل شدند (بلافاصله زیر «Hosts») و هر کدام آدرس خود را دارند — `/outbound` و `/routing`. پیش‌تر مسیریابی داخل زیرمنوی «پیکربندی‌های Xray» باز می‌شد و outbounds به‌عنوان زبانه‌ای از صفحه Xray. در زیرمنوی «پیکربندی‌های Xray» اکنون تنها این موارد باقی می‌مانند: اصلی، متعادل‌کننده، DNS و قالب پیشرفته. لینک‌های مستقیم به `/outbound` و `/routing` و بارگذاری مجدد صفحه طبق انتظار کار می‌کنند.
- **قوانین مسیریابی را می‌توان با کلید فعال و غیرفعال کرد** — یک قانون مسیریابی مجزا را اکنون می‌توان به‌طور موقت با یک کلید **غیرفعال** کرد، بدون حذف آن. در جدول قوانین ستون **«فعال»** با کلید وجود دارد، در فرم قانون نیز فیلد «فعال» یک کلید است. قانون غیرفعال‌شده در پیکربندی نهایی Xray قرار نمی‌گیرد. قانون سرویسی آمار (`api`) را نمی‌توان غیرفعال کرد — کلید آن قفل است.
- **صادرات قوانین مسیریابی و outbounds در پنجره مودال پیش‌نمایش باز می‌شود** — دکمه **«صادرات»** قوانین مسیریابی و خروجی‌ها اکنون به‌جای دانلود فوری فایل، یک پنجره مودال با پیش‌نمایش JSON و دکمه‌های **«کپی»** و **«دانلود»** باز می‌کند. در زبانه «مسیریابی» «واردات» و «صادرات» در منوی کشویی **«بیشتر»** گرد آمده‌اند (مانند زبانه Outbounds).
- **تست همه خروجی‌ها اکنون outbounds از اشتراک‌ها را نیز بررسی می‌کند؛ direct/dns دیگر تست نمی‌شوند** — دکمه **«تست همه»** در صفحه «خروجی‌ها» اکنون outbounds دریافت‌شده از اشتراک‌ها (جدول «از اشتراک‌ها») را نیز بررسی می‌کند — سطرهای آن‌ها هم با نتیجه برجسته می‌شوند. در عین حال outbounds با نوع `freedom` («direct») و `dns` دیگر در هیچ حالتی تست نمی‌شوند (این‌ها پراکسی نیستند): دکمه تست برای آن‌ها در دسترس نیست و «تست همه» آن‌ها را رد می‌کند.
- **FinalMask: آرایه‌های Lengths/Delays به‌ازای هر بخش به‌جای Length/Delay تکی** — در ماسک fragment (FinalMask) فیلدهای تکی Length و Delay با فهرست‌های Lengths و Delays جایگزین شدند: برای هر سگمنت می‌توان بازه جداگانه‌ای از طول (مثلاً 100-200) و تأخیر بر حسب میلی‌ثانیه (مثلاً 10-20 یا 0) تعیین کرد. سطرها را می‌توان افزود و حذف کرد؛ مقادیر قبلاً ذخیره‌شده به‌صورت خودکار منتقل می‌شوند.
- **Loopback outbound: بلوک Sniffing افزوده شد** — به outbound از نوع loopback بلوک Sniffing با همان پارامترهای inbound افزوده شد: فعال‌سازی، destOverride، Metadata Only، Route Only و فهرست دامنه‌های مستثنی‌شده.
- **Hysteria2 / Salamander: حالت Gecko (packetSize) و TLS برای ماسک Realm** — در ماسک UDP (FinalMask) برای Hysteria2 امکانات گسترش یافت. به ماسک Salamander انتخابگر Mode افزوده شد: حالت Gecko پدینگ تصادفی بسته‌ها را با فیلدهای Min/Max اندازه (از 1 تا 2048، به‌صورت پیش‌فرض 512-1200) برای محافظت در برابر تحلیل طول بسته‌ها اضافه می‌کند. به ماسک Realm یک بلوک اختیاری TLS Config افزوده شد: Server Name (SNI)، ALPN (h3/h2/http/1.1)، Fingerprint و Allow Insecure.
- **واردات لینک share به outbound تنظیمات xmux را حفظ می‌کند** — هنگام واردات outbound از لینک share اکنون تنظیمات مالتی‌پلکسر **xmux** (XHTTP) حفظ می‌شوند: پیش‌تر بی‌صدا از دست می‌رفتند. پس از واردات، مقادیر در زیرفرم XMUX جای‌گذاری می‌شوند.
- **تگ‌های outbounds از اشتراک‌ها کاراکترهای غیر-ASCII (سیریلیک) را حفظ می‌کنند** — تگ‌های outbounds دریافت‌شده از اشتراک‌ها اکنون کاراکترهای غیر-ASCII (مثلاً سیریلیک) را حفظ می‌کنند و خوانا می‌مانند، نه اینکه به ارقام تنها تقلیل یابند.

### 12. نودها (مولتی‌پنل، master/slave)
- **نودها: حالت جدید بررسی TLS — Mutual TLS (گواهی کلاینت)** — در فرم نود، حالت بررسی TLS اکنون چهار گزینه دارد: Verify (CA سیستم)، Pin (تثبیت گواهی بر اساس SHA-256)، Skip (بدون بررسی) و حالت جدید Mutual TLS (گواهی کلاینت). در حالت Mutual TLS پنل افزون بر آن، خود را با یک گواهی کلاینت صادرشده توسط CA خودش به نود تأیید می‌کند؛ توکن API برای چنین نودی اختیاری می‌شود. برای فعال‌سازی mTLS: روی نود حالت Mutual TLS را تعیین کنید، CA پنل مدیریت‌کننده را از بخش Node mTLS کپی کنید، آن را روی نود به‌عنوان CA والد مورد اعتماد ثبت کنید و نود را راه‌اندازی مجدد کنید.
- **نودها: بخش «Node mTLS» — کپی CA پنل و CA والد مورد اعتماد** — در صفحه نودها بخش Node mTLS برای پیکربندی احراز هویت متقابل TLS بین پنل‌ها افزوده شد. دکمه «کپی CA این پنل» گواهی ریشه پنل را در کلیپ‌بورد کپی می‌کند — باید آن را به نودهای مدیریت‌شده‌ای داد که گواهی کلاینت پنل را بررسی خواهند کرد. فیلد «CA والد مورد اعتماد» زمانی استفاده می‌شود که خود پنل یک نود باشد: CA پنل مدیریت‌کننده را اینجا جای‌گذاری کنید تا گواهی کلاینت آن الزامی شود و پنل را راه‌اندازی مجدد کنید. TLS متقابل به‌دلخواه فعال می‌شود؛ اگر فیلدها خالی باشند، نودها طبق طرح پیشین کار می‌کنند — تنها با توکن API.
- **مسیریابی اتصال خروجی پنل به نود (Connection outbound)** — در فرم نود فیلد **«Connection outbound»** (اتصال خروجی) افزوده شد. اگر در آن یک تگ Xray-outbound انتخاب شود، ترافیک پنل به API آن نود از طریق outbound تعیین‌شده می‌رود (پنل خودش یک bridge-inbound روی loopback به پیکربندی کاری اضافه کرده و آن را بدون راه‌اندازی مجدد اعمال می‌کند). مقدار خالی = اتصال مستقیم. در فهرست، تگ‌ها به «خروجی‌ها» و «متعادل‌کننده‌ها» گروه‌بندی شده‌اند و خروجی‌های blackhole پنهان‌اند.
- **نودها: مسیریابی ترافیک پنل←نود از طریق outbound انتخاب‌شده («Connection outbound»)** — در فرم نود فیلد «Connection outbound» افزوده شد: امکان هدایت ترافیک درخواست‌های پنل به نود از طریق Xray outbound انتخاب‌شده را می‌دهد (outbounds معمولی و متعادل‌کننده‌ها در دسترس‌اند). پنل به‌صورت خودکار یک loopback-bridge inbound به پیکربندی کاری اضافه می‌کند و تغییرات را بدون راه‌اندازی مجدد اعمال می‌کند. برای اتصال مستقیم، فیلد را خالی بگذارید.
- **نودها: حذف نود مادامی‌که inbounds به آن پیوند خورده‌اند مسدود است** — حذف نود تنها پس از آن ممکن است که همه inbounds از آن برداشته شوند. اگر هنوز دست‌کم یک inbound به نود پیوند خورده باشد، پنل حذف را با خطا رد می‌کند — ابتدا این inbounds را جدا یا حذف کنید، سپس نود را حذف کنید.
- **نودها: در صفحه نودها سرعت زنده inbound‌های مستقر روی نود نمایش داده می‌شود** — در صفحه نودها برای inbound‌های مستقر روی نود، اکنون کلاینت‌های آنلاین، شمارنده‌ها و سرعت جاری انتقال نمایش داده می‌شوند. چیپ «پایان‌یافته» (ended) تنها کلاینت‌های منقضی‌شده و دارای ترافیک تمام‌شده را می‌شمارد (غیرفعال‌شده‌ها دیگر در آن لحاظ نمی‌شوند).

### 14. ربات Telegram
- **اعلان‌ها: گذرگاه رویدادی جدید با کانال‌های Telegram و Email (SMTP)** — سامانه اعلان‌های رویدادی با دو کانال تحویل — Telegram و Email — افزوده شد. در زبانه اعلان‌ها، رویدادها در قالب کارت گروه‌بندی شده‌اند: Outbound (افت/بازیابی)، Xray Core (خاتمه اضطراری)، Nodes (نود در دسترس نیست/بازیابی شد)، System (بار بالای CPU و حافظه با آستانه قابل‌تنظیم بر حسب %)، Security (تلاش ورود). هر گروه یک کلید اصلی و شمارنده رویدادهای انتخاب‌شده دارد. مجموعه رویدادهای فعال‌شده به‌صورت جداگانه برای Telegram و Email تنظیم می‌شود؛ به‌صورت پیش‌فرض «تلاش ورود» و «بار بالای CPU» فعال‌اند.
- **اعلان‌ها: کانال جدید Email/SMTP و تنظیمات سرور SMTP** — کانال جدید اعلان از طریق ایمیل با SMTP افزوده شد. در زبانه تنظیمات SMTP این موارد تعیین می‌شوند: فعال‌سازی اعلان‌های ایمیل، هاست و پورت SMTP (به‌صورت پیش‌فرض 587)، نام کاربری، رمز عبور (مخفی ذخیره می‌شود)، فهرست گیرندگان (جداشده با ویرگول) و نوع رمزنگاری — none، STARTTLS (پیش‌فرض) یا TLS. دکمه «ارسال ایمیل آزمایشی» اتصال را بررسی می‌کند و نشان می‌دهد خطا در کدام مرحله (اتصال، احراز هویت، ارسال) رخ داده است. در زبانه دوم رویدادهایی که برای آن‌ها ایمیل ارسال می‌شود انتخاب می‌گردند.
- **اعلان‌ها: هشدار بار بالای حافظه (آستانه RAM)** — به هشدارهای بار بالای CPU، هشدار بار بالای حافظه افزوده شد. در گروه رویدادهای «System» گزینه «Memory high (%)» با فیلد آستانه مخصوص خود (به‌صورت پیش‌فرض 80%) ظاهر شد؛ پنل هر دقیقه یک‌بار بار RAM را بررسی می‌کند و در صورت عبور از آستانه، اعلانی به کانال‌های انتخاب‌شده ارسال می‌کند.

### 15. پایگاه‌های جغرافیایی (geoip / geosite و سفارشی)
- **به‌روزرسانی پایگاه‌های جغرافیایی: وضعیت برای هر فایل و رد کردن راه‌اندازی مجدد بدون تغییر** — به‌روزرسانی پایگاه‌های جغرافیایی (geoip/geosite، شامل مجموعه‌های IR و RU) اکنون وضعیت هر فایل را نمایش می‌دهد: به‌روزشده، از پیش به‌روز است یا خطای دانلود. راه‌اندازی مجدد Xray (و در نتیجه قطع اتصالات فعال) تنها زمانی رخ می‌دهد که دست‌کم یک فایل واقعاً به‌روز شده باشد؛ در نبود تغییر، پنل راه‌اندازی مجدد نمی‌شود. همین رفتار برای دستور x-ui update-all-geofiles وجود دارد.

### 16. بهره‌برداری: پشتیبان‌ها، لاگ‌ها، به‌روزرسانی، CLI
- **محدودیت IP کلاینت تنها در صورت نصب fail2ban کار می‌کند؛ در غیر این صورت فیلد قفل می‌شود** — محدودیت تعداد IP کلاینت اکنون تنها در صورتی اعمال می‌شود که روی سرور fail2ban نصب باشد. اگر نصب نباشد، فیلد «IP Limit» در فرم کلاینت و هنگام افزودن گروهی غیرفعال شده و راهنمایی توضیحی نمایش داده می‌شود (روی Windows — با پیامی جداگانه)، و محدودیت‌های قبلاً تعیین‌شده روی چنین سرورهایی به‌صورت خودکار صفر می‌شوند، چون به‌هرحال اعمال نمی‌شدند. مسدودسازی fail2ban اکنون هم به TCP و هم به UDP گسترش می‌یابد.
- **نصب خودکار fail2ban هنگام نصب و به‌روزرسانی پنل** — هنگام نصب و به‌روزرسانی پنل روی یک سرور معمولی، fail2ban اکنون به‌صورت خودکار نصب و پیکربندی می‌شود (پیش‌تر این تنها در Docker رخ می‌داد)، تا قابلیت محدودیت IP بدون تنظیم اضافی کار کند. این رفتار با متغیر محیطی XUI_ENABLE_FAIL2BAN کنترل می‌شود: پیکربندی انجام می‌شود اگر متغیر تعیین نشده باشد یا برابر true باشد. اجرای دستی با دستور x-ui setup-fail2ban در دسترس است؛ خطای fail2ban نصب یا به‌روزرسانی را متوقف نمی‌کند.
- **بازنویسی پورت پنل از طریق متغیر XUI_PORT** — متغیر محیطی XUI_PORT افزوده شد که پورت پنل وب را تنها برای مدت کارکرد فرایند جاری تعیین می‌کند، بدون تغییر مقدار ذخیره‌شده webPort در پایگاه داده. مقادیر از 1 تا 65535 مجازند؛ مقدار خالی، نادرست یا خارج از بازه نادیده گرفته می‌شود (از webPort استفاده می‌شود) با هشداری در لاگ. هنگام استفاده از Docker با شبکه bridge، پورت منتشرشده کانتینر باید با XUI_PORT منطبق باشد، مثلاً XUI_PORT=8080 و ports: «8080:8080».
- **CLI: پرچم‌های -webCert/-webCertKey اکنون در زیردستور setting اعمال می‌شوند** — در CLI پرچم‌های -webCert و -webCertKey اکنون در زیردستور x-ui setting نیز کار می‌کنند (پیش‌تر بی‌صدا نادیده گرفته می‌شدند و پنل بدون HTTPS می‌ماند). با تعیین آن‌ها می‌توان بی‌درنگ مسیر گواهی و کلید پنل وب را تنظیم کرد، بدون فراخوانی زیردستور جداگانه cert.
- **نام فایل پشتیبان پایگاه داده بر اساس آدرس سرور ساخته می‌شود** — فایل‌های پشتیبان پایگاه داده اکنون به‌جای x-ui.db / x-ui.dump ثابت، بر اساس آدرس سرور نام‌گذاری می‌شوند. هنگام دانلود از مرورگر، نام از آدرس پنل در نوار آدرس گرفته می‌شود، در غیر این صورت — از وب‌دامنه تنظیم‌شده، و در نبود آن — از IP عمومی (ابتدا IPv4، سپس IPv6). به این ترتیب پشتیبان‌ها از سرورهای مختلف به‌راحتی قابل تمایزند. پسوند برای SQLite همچنان .db و برای PostgreSQL همان .dump باقی می‌ماند.
- **پشتیبانی از نصب و به‌روزرسانی روی هاست‌های فقط IPv6** — اسکریپت‌های نصب و به‌روزرسانی اکنون روی سرورهای فقط IPv6 به‌درستی کار می‌کنند: دانلود رلیز و فایل‌های کمکی دیگر به‌اجبار از IPv4 استفاده نمی‌کند، بنابراین پنل را می‌توان روی هاست بدون آدرس IPv4 نصب و به‌روز کرد.
