## Що нового у 3.4.0

Цей розділ коротко перелічує зміни версії **3.4.0** відносно 3.3.1, які видно користувачу панелі, згруповані за розділами посібника. Подробиці щодо кожної функції — у відповідному розділі нижче.

### 3. Огляд / Дашборд
- **Історія системних метрик: нові інтервали агрегації 12h/24h/48h** — У вікні історії системних метрик до інтервалів усереднення додано значення 12h, 24h та 48h — тепер графіки (CPU, RAM, трафік, пакети, з’єднання, диск, онлайн, навантаження) можна переглядати за період до двох діб.

### 4. Inbounds: створення та загальні параметри
- **Inbound: попередження про конфлікт із зарезервованим портом Xray API** — Під час створення або зміни inbound панель тепер не дає зайняти зарезервований порт внутрішнього Xray API (за замовчуванням 62789 на 127.0.0.1): локальний TCP-inbound на цьому порту на loopback відхиляється з помилкою конфлікту порту. На вузлах (nodes) обмеження не діє — у них власний Xray.
- **Tunnel/TProxy: приймання streamSettings без ключа security** — Inbound типу tunnel/TProxy, у яких streamSettings не містять блоку security, тепер відкриваються та зберігаються без помилки валідації.
- **Inbounds: колонка Speed (жива швидкість) у списку** — У списку inbounds з’явилася колонка Speed з поточною швидкістю трафіку (віддача/завантаження) за кожним inbound.

### 5. Протоколи
- **Shadowsocks-2022: перегенерація клієнтських PSK при зміні методу з іншим розміром ключа** — Для Shadowsocks-2022: при зміні методу шифрування на метод з іншим розміром ключа (наприклад між aes-256 та aes-128) клієнтські PSK тепер автоматично перегенеровуються під нову довжину при збереженні inbound. Наслідок: задіяним клієнтам потрібно заново отримати підписку (старі посилання перестануть працювати).
- **WireGuard: видалено поле workers** — З форм WireGuard (inbound та outbound) прибрано поле workers: xray-core v26.6.22 більше його не використовує. Раніше збережені конфіги працюють без змін — поле просто ігнорується.
- **VLESS+XHTTP: флоу xtls-rprx-vision у посиланнях і підписках** — Для VLESS поверх XHTTP флоу xtls-rprx-vision тепер коректно потрапляє в посилання та підписки (зокрема для XHTTP+REALITY і у форматі Clash/Mihomo). Раніше панель зберігала flow, але клієнти отримували конфігурацію без vision.

### 6. Транспорт (Stream Settings)
- **XHTTP: перейменування sessionID-полів + Session ID Table / Length** — У транспорті XHTTP поля сесії перейменовано: Session ID Placement та Session ID Key (раніше — Session Placement / Session Key). Додано два параметри. Session ID Table — набір символів для генерації ідентифікаторів сесії: можна вибрати визначений наперед (ALPHABET, Base62, hex, number та ін.) або ввести довільний ASCII-рядок; порожньо — значення за замовчуванням xray-core. Session ID Length — довжина або діапазон (наприклад 8-16) згенерованих ідентифікаторів; враховується лише коли задано Session ID Table, мінімум має бути більшим за 0. Старі збережені inbound мігруються автоматично.
- **Inbound: пресет Real client IP для визначення реального IP за CDN/релеєм** — У налаштуваннях сокета (sockopt) inbound з’явився вибір Real client IP — пресет для визначення реального IP відвідувача, коли трафік приходить через CDN або релей (інакше фіксується адреса посередника). Доступні три варіанти: Off / direct (без обробки), Cloudflare CDN (довіряти X-Forwarded-For) та L4 relay / Spectrum (PROXY) (приймати заголовок PROXY-протоколу). Пресети взаємовиключні та попереджають, якщо обраний транспорт їх не підтримує. Ці поля ніколи не надсилаються клієнтам у підписках.
- **gRPC: заголовок Trusted X-Forwarded-For тепер враховується** — Заголовок Trusted X-Forwarded-For тепер враховується і на gRPC-транспорті (раніше — лише WebSocket, HTTPUpgrade та XHTTP). Для gRPC-inbound панель більше не показує попередження про непідтримуваний заголовок.

### 7. Безпека з’єднання: TLS, XTLS та REALITY
- **TLS: нові поля Verify Peer Cert By Name, Curve Preferences, Master Key Log, ECH Sockopt** — Verify Peer Cert By Name — імена (через кому), за якими клієнт перевіряє сертифікат сервера замість SNI; сучасна заміна видаленого allowInsecure, передається в посиланнях та підписках, на сервер не пишеться. Curve Preferences — обмеження та порядок кривих обміну ключами TLS (наприклад X25519MLKEM768, X25519); порожньо — значення за замовчуванням. Master Key Log — шлях для запису TLS-ключів (формат SSLKEYLOGFILE) для налагодження у Wireshark; у проді залишати порожнім. ECH Sockopt — параметри сокета для отримання ECH-конфігурації (dialerProxy, Domain Strategy, TCP Fast Open, Multipath TCP).
- **REALITY: обмеження швидкості fallback (Limit Fallback) та Master Key Log** — Для кожного напрямку (Upload та Download) задаються: After Bytes — скільки байтів пропустити на повній швидкості перед початком обмеження (0 — обмежувати з першого байта); Bytes Per Sec — стеля швидкості fallback-трафіку, щоб зонди не використовували сервер як безкоштовний канал (0 — без обмеження, вимикає напрямок); Burst Bytes Per Sec — запас для короткочасних сплесків. Також додано поле Master Key Log (шлях до файлу SSLKEYLOGFILE для налагодження).
- **TLS: кнопки заповнення Pinned Peer Cert SHA-256 із сертифіката inbound і за SNI** — Поряд з полем Pinned Peer Cert SHA-256 тепер дві кнопки автозаповнення замість попередньої кнопки випадкового хешу. Перша підставляє SHA-256 сертифіката самого inbound. Друга отримує хеш живого сертифіката сервера, виконавши TLS-підключення за вказаним SNI (serverName має бути заповнений). Отримані хеші додаються в поле (через кому) і потрапляють у посилання для закріплення сертифіката в клієнта.
- **TLS: OCSP-stapling за замовчуванням вимкнено для нових сертифікатів inbound** — Для нових inbound OCSP stapling за замовчуванням вимкнено (інтервал 0). Це прибирає помилки в логах xray для сертифікатів без OCSP-респондера (наприклад Let's Encrypt). Поле залишилося — його можна увімкнути для сертифікатів, які підтримують stapling.
- **REALITY: сумісність із полем dest (аліас target)** — Якщо REALITY-inbound було створено з полем dest (старими версіями панелі, через API або зовнішніми інструментами), тепер він коректно завантажується: значення dest підставляється в поле Target. Раніше Target виявлявся порожнім, і пересохранення ламало REALITY.

### 8. Клієнти
- **Вкладка «Links» у редакторі клієнта (зовнішні посилання та підписки)** — На ній кнопкою **Add External Link** додають сторонні share-посилання (`vless://`, `vmess://`, `trojan://`, `ss://`, `hysteria2://`, `wireguard://`), а кнопкою **Add External Subscription** — URL віддалених підписок. Усе перелічене підмішується у видачу підписки цього клієнта (формати raw, JSON та Clash): посилання додаються як є, а віддалені підписки панель періодично завантажує та об’єднує їхні конфігурації з власними.
- **Поле «Ліміт IP» тепер вимикається без Fail2ban** — Поле **Ліміт IP** тепер працює лише за встановленого та активного Fail2ban. Якщо Fail2ban не встановлено (або система Windows, або функцію вимкнено на сервері), поле редактора клієнта блокується, а при наведенні показується підказка з пропозицією встановити Fail2ban з bash-меню `x-ui`. Під час оновлення панелі у клієнтів на серверах без Fail2ban збережений ліміт IP обнуляється, оскільки він там все одно не застосовувався.
- **Видалення непов’язаних клієнтів, експорт та імпорт клієнтів** — У меню **Ще** на сторінці **Клієнти** додано три операції. **Експорт клієнтів** показує JSON-список усіх клієнтів (у форматі `{client, inboundIds}`) з кнопками копіювання та завантаження (`clients-export.json`). **Імпорт клієнтів** приймає такий самий JSON: клієнти з прив’язками відтворюються та прив’язуються до inbound, клієнти без прив’язок відновлюються як окремі записи, а вже наявні email не перезаписуються (вони позначаються як пропущені). **Видалити непов’язаних клієнтів** видаляє всіх клієнтів, не прив’язаних до жодного inbound, разом з їхнім трафіком, журналом IP та зовнішніми посиланнями; дія незворотна.
- **Журнал IP клієнта: час підключення та ім’я ноди** — У журналі IP клієнта (кнопка перегляду поряд з полем «Ліміт IP» та в картці «Інформація про клієнта») кожен запис тепер містить, окрім самого IP, час останнього звернення та мітку ноди (`@ ім'я_ноди`), через яку зафіксовано підключення, — у мультипанельній конфігурації видно, через який вузол підключався клієнт.
- **Скидання мітки групи в клієнта в одиночному редакторі** — Тепер якщо в редакторі одного клієнта очистити поле **Група** та зберегти, мітка групи коректно знімається — раніше клієнт міг продовжувати відображатися під попередньою групою до повторного збереження.
- **Список клієнтів оновлюється автоматично (фонове опитування)** — Список клієнтів тепер оновлюється автоматично: панель раз на кілька секунд підтягує актуальну сторінку, тому щойно підключені клієнти та змінений порядок сортування з’являються без ручного оновлення.

### 10. Підписки (Subscription)
- **Керовані Hosts: перевизначення subscription-посилань за хостами** — У версії 3.4.0 додано розділ Hosts (пункт бічного меню). Для кожного inbound можна задати один або кілька Host-ендпоінтів, які підставляються в subscription-посилання замість адреси, порту та TLS-параметрів самого inbound — це зручно для роздачі трафіку через CDN або релей. У хоста задаються Remark та опис, прив’язка до inbound, Address (порожньо — успадковується адреса inbound) і Port (0 — успадковується порт inbound), параметри Security (same/tls/none/reality), а також Host header, Path, Mux, Sockopt, Final Mask, виключення з форматів підписки (raw/json/clash) та параметри для Clash/mihomo. Хости впорядковуються всередині inbound і підтримують масові операції.
- **Remark Template замінив конструктор моделі ремарки; змінні {{VAR}}** — Попередній конструктор імені профілю (вибір Inbound/Email/External Proxy та роздільника) замінено полем «Remark Template». У ньому ви пишете довільний формат імені, вставляючи змінні кнопкою: ідентифікація клієнта ({{EMAIL}}, {{INBOUND}}, {{HOST}}, {{ID}}, {{SUB_ID}}, {{COMMENT}}, {{TELEGRAM_ID}}), трафік ({{TRAFFIC_USED}}, {{TRAFFIC_LEFT}}, {{TRAFFIC_TOTAL}}, {{UP}}, {{DOWN}}, {{USAGE_PERCENTAGE}}) і строк/статус ({{DAYS_LEFT}}, {{TIME_LEFT}}, {{EXPIRE_DATE}}, {{JALALI_EXPIRE_DATE}}, {{STATUS}}, {{STATUS_EMOJI}}). Змінні підставляються індивідуально для кожного клієнта під час генерації підписки, доступний попередній перегляд. Сегменти, розділені символом «|», з безлімітними значеннями приховуються автоматично, а відомості про витрату та строк показуються лише на першому посиланні клієнта. Якщо поле залишити порожнім, використовується попередня модель ремарки.
- **Per-client зовнішні посилання та віддалені підписки (вкладка Links)** — Тут для окремого клієнта можна прикріпити сторонні share-посилання (Add External Link) та адреси зовнішніх підписок (Add External Subscription) — вони будуть включені в його власну підписку (формати RAW, JSON та Clash). Зовнішні підписки завантажуються панеллю та об’єднуються з конфігураціями клієнта. Це зручно, щоб видати клієнту додаткові сервери поверх ваших inbound.
- **Happ: приховування налаштувань сервера в клієнті (Hide server settings)** — На вкладці Happ налаштувань підписки додано перемикач «Hide server settings». Коли його ввімкнено, у клієнті Happ приховується можливість переглядати та змінювати параметри сервера. Опція діє лише для клієнта Happ.
- **Ім’я вузла більше не дописується до імені профілю в підписці** — Ім’я вузла (Node) більше не додається до імені профілю в підписці. У клієнтському застосунку показується лише ремарка inbound, задана адміністратором, без внутрішнього суфікса виду «@ім’я-вузла».
- **Мітку моделі ремарки перейменовано Other → External Proxy (потім замінено шаблоном)** — Окремо документувати не потрібно: перейменування пункту моделі ремарки «Other» на «External Proxy» поглинуто переходом на нове поле Remark Template, де конструктор моделі з UI прибрано.
- **Коректність посилань підписки: SS2022, Shadowrocket, SIP002 obfs, XHTTP у Clash** — Покращено сумісність генерованих посилань підписки: виправлено кодування SS2022, deep-link для Shadowrocket, видачу Shadowsocks+obfs у форматі SIP002 (плагін obfs-local) та повний набір полів XHTTP у підписках Clash/Mihomo. Окремих налаштувань не потрібно — посилання просто коректніше розпізнаються клієнтами.
- **Модель ремарок підписки: пункт «Other» перейменовано на «External Proxy»** — У налаштуваннях підписки в моделі ремарок пункт **«Other»** перейменовано на **«External Proxy»** (джерело — ремарка зовнішнього проксі). Поведінка не змінилася, наявні налаштування залишаються сумісними.
- **Підписки: вибір змінних ремарки кліком по чипах (Remark variable picker)** — Поряд з полем Remark Template доступний згрупований набір змінних-чипів: клік по змінній {{VAR}} вставляє її в шаблон, при наведенні показується опис. У полях ремарки та імені хоста допускається й спрощений запис в одинарних дужках — {DATA_LEFT}, {EXPIRE_DATE}, {PROTOCOL}, {TRANSPORT} тощо; панель автоматично перетворює його у внутрішній формат {{...}}.

### 11. Xray: маршрутизація, outbounds, DNS та розширення
- **Маршрутизацію та Outbounds винесено в окремі пункти бічного меню** — Починаючи з цієї версії **«Вихідні» (Outbounds)** та **«Маршрутизація» (Routing)** винесено в окремі пункти бічного меню (одразу під «Хости»), у кожного своя адреса — `/outbound` та `/routing`. Раніше маршрутизація відкривалася всередині підменю «Конфігурації Xray», а outbounds — як вкладка сторінки Xray. У підменю «Конфігурації Xray» тепер залишаються лише: Основне, Балансувальник, DNS та Розширений шаблон. Прямі посилання на `/outbound` та `/routing` і перезавантаження сторінки працюють як очікується.
- **Правила маршрутизації можна вмикати та вимикати перемикачем** — Окреме правило маршрутизації тепер можна тимчасово **вимкнути** перемикачем, не видаляючи його. У таблиці правил є стовпець **«Увімкнути»** з перемикачем, у формі правила поле «Увімкнути» — теж перемикач. Вимкнене правило не потрапляє в підсумкову конфігурацію Xray. Службове правило статистики (`api`) вимкнути не можна — його перемикач заблоковано.
- **Експорт правил маршрутизації та outbounds відкривається в модальному вікні попереднього перегляду** — Кнопка **«Експорт»** правил маршрутизації та вихідних тепер не завантажує файл одразу, а відкриває модальне вікно з попереднім переглядом JSON і кнопками **«Копіювати»** та **«Завантажити»**. На вкладці «Маршрутизація» «Імпорт» та «Експорт» зібрано у випадне меню **«ще»** (як на вкладці Outbounds).
- **Тест усіх вихідних тепер перевіряє й outbounds із підписок; direct/dns більше не тестуються** — Кнопка **«Тест усіх»** на сторінці «Вихідні» тепер перевіряє й outbounds, отримані з підписок (таблиця «з підписок») — їхні рядки також підсвічуються результатом. При цьому outbounds `freedom` («direct») та `dns` більше не тестуються в жодному режимі (це не проксі): кнопка тесту в них недоступна, а «Тест усіх» їх пропускає.
- **FinalMask: пофрагментні масиви Lengths/Delays замість одиночних Length/Delay** — У масці fragment (FinalMask) одиночні поля Length та Delay замінено списками Lengths та Delays: для кожного сегмента можна задати окремий діапазон довжини (наприклад 100-200) та затримки в мс (наприклад 10-20 або 0). Рядки можна додавати та видаляти; раніше збережені значення переносяться автоматично.
- **Loopback outbound: додано блок Sniffing** — У outbound типу loopback з’явився блок Sniffing з тими самими параметрами, що й у inbound: увімкнення, destOverride, Metadata Only, Route Only та список виключених доменів.
- **Hysteria2 / Salamander: режим Gecko (packetSize) та TLS для маски Realm** — У масці UDP (FinalMask) для Hysteria2 розширено можливості. У маски Salamander додано селектор Mode: режим Gecko додає випадкове доповнення пакетів з полями Min/Max розміру (від 1 до 2048, за замовчуванням 512-1200) для захисту від аналізу довжини пакетів. У маски Realm з’явився необов’язковий блок TLS Config: Server Name (SNI), ALPN (h3/h2/http/1.1), Fingerprint та Allow Insecure.
- **Імпорт share-посилання в outbound зберігає налаштування xmux** — Під час імпорту outbound із share-посилання тепер зберігаються налаштування мультиплексора **xmux** (XHTTP): раніше вони мовчки втрачалися. Після імпорту значення підставляються у підформу XMUX.
- **Теги outbounds із підписок зберігають не-ASCII символи (кирилиця)** — Теги outbounds, отриманих із підписок, тепер зберігають не-ASCII символи (наприклад, кирилицю) та залишаються читабельними, а не зводяться до самих цифр.

### 12. Вузли (мультипанель, master/slave)
- **Вузли: новий режим перевірки TLS — Mutual TLS (клієнтський сертифікат)** — У формі вузла режим перевірки TLS тепер має чотири варіанти: Verify (системний CA), Pin (фіксація сертифіката за SHA-256), Skip (без перевірки) та новий Mutual TLS (клієнтський сертифікат). У режимі Mutual TLS панель додатково підтверджує себе вузлу клієнтським сертифікатом, випущеним її власним CA; API-токен для такого вузла стає необов’язковим. Щоб увімкнути mTLS: на вузлі задайте режим Mutual TLS, скопіюйте CA керуючої панелі з розділу Node mTLS, пропишіть його на вузлі як довірений батьківський CA та перезапустіть вузол.
- **Вузли: розділ «Node mTLS» — копіювання CA панелі та довірений батьківський CA** — На сторінці Вузли додано розділ Node mTLS для налаштування взаємної TLS-автентифікації між панелями. Кнопка «Скопіювати CA цієї панелі» копіює кореневий сертифікат панелі в буфер обміну — його потрібно передати керованим вузлам, які перевірятимуть клієнтський сертифікат панелі. Поле «Довірений батьківський CA» використовується, коли сама панель є вузлом: вставте сюди CA керуючої панелі, щоб вимагати її клієнтський сертифікат, і перезапустіть панель. Взаємний TLS вмикається за бажанням; якщо поля порожні, вузли працюють за попередньою схемою — лише з API-токеном.
- **Маршрутизація вихідного підключення панелі до вузла (Connection outbound)** — У формі вузла з’явилося поле **«Connection outbound»** (вихідне підключення). Якщо вибрати в ньому тег Xray-outbound, трафік панелі до API цього вузла піде через вказаний outbound (панель сама додасть у робочу конфігурацію міст-inbound на loopback і застосує його без перезапуску). Порожнє значення = пряме підключення. У списку теги згруповано на «Вихідні» та «Балансувальники», blackhole-вихідні приховано.
- **Вузли: маршрутизація трафіку панель→вузол через обраний outbound («Connection outbound»)** — У формі вузла з’явилося поле «Connection outbound»: воно дозволяє спрямувати трафік звернень панелі до вузла через обраний Xray outbound (доступні звичайні outbounds та балансувальники). Панель автоматично додає в робочу конфігурацію loopback-bridge inbound і застосовує зміни без перезапуску. Залиште поле порожнім для прямого підключення.
- **Вузли: видалення вузла блокується, поки до нього прив’язані inbounds** — Видалити вузол можна лише після того, як з нього знято всі inbounds. Якщо до вузла все ще прив’язано хоча б один inbound, панель відхилить видалення з помилкою — спочатку відкріпіть або видаліть ці inbounds, потім видаліть вузол.
- **Вузли: на сторінці вузлів показується жива швидкість inbound'ів, розміщених на вузлі** — На сторінці Вузли для inbound'ів, розміщених на вузлі, тепер показуються онлайн-клієнти, лічильники та поточна швидкість передачі. Чип «завершені» (ended) рахує лише клієнтів, що минули строк та вичерпали трафік (вимкнені до нього більше не входять).

### 14. Telegram-бот
- **Сповіщення: нова шина подій з каналами Telegram та Email (SMTP)** — Додано систему подійних сповіщень з двома каналами доставки — Telegram та Email. На вкладці сповіщень події згруповано картками: Outbound (падіння/відновлення), Xray Core (аварійне завершення), Nodes (вузол недоступний/відновлений), System (високе навантаження CPU та пам’яті з налаштовуваним порогом у %), Security (спроба входу). У кожної групи є майстер-перемикач і лічильник вибраних подій. Набір увімкнених подій налаштовується окремо для Telegram та Email; за замовчуванням увімкнено «спробу входу» та «високе навантаження CPU».
- **Сповіщення: новий канал Email/SMTP та налаштування SMTP-сервера** — Додано новий канал сповіщень електронною поштою через SMTP. На вкладці налаштувань SMTP задаються: увімкнення email-сповіщень, хост і порт SMTP (за замовчуванням 587), ім’я користувача, пароль (зберігається приховано), список одержувачів (через кому) та тип шифрування — none, STARTTLS (за замовчуванням) або TLS. Кнопка «Надіслати тестового листа» перевіряє з’єднання та показує, на якому етапі (підключення, автентифікація, надсилання) виникла помилка. На другій вкладці вибираються події, про які надходитимуть листи.
- **Сповіщення: оповіщення про високе навантаження пам’яті (поріг RAM)** — До оповіщень про високе навантаження CPU додано оповіщення про високе навантаження оперативної пам’яті. У групі подій «System» з’явилося «Memory high (%)» з власним полем порогу (за замовчуванням 80%); панель раз на хвилину перевіряє навантаження RAM і при перевищенні порогу надсилає сповіщення в обрані канали.

### 15. Гео-бази (geoip / geosite та кастомні)
- **Оновлення гео-баз: статус за кожним файлом та пропуск перезапуску без змін** — Оновлення гео-баз (geoip/geosite, включно з наборами IR та RU) тепер показує статус за кожним файлом: оновлено, вже актуальний або помилка завантаження. Перезапуск Xray (а отже й розрив активних з’єднань) відбувається лише якщо хоча б один файл дійсно було оновлено; за відсутності змін панель не перезапускається. Така сама поведінка в команди x-ui update-all-geofiles.

### 16. Експлуатація: бекапи, логи, оновлення, CLI
- **IP-ліміт клієнта працює лише при встановленому fail2ban; поле блокується інакше** — Обмеження за кількістю IP у клієнта тепер діє лише якщо на сервері встановлено fail2ban. Якщо його немає, поле «IP Limit» у формі клієнта та при масовому додаванні стає недоступним з пояснювальною підказкою (на Windows — окремим повідомленням), а раніше задані ліміти на таких серверах автоматично обнуляються, оскільки вони все одно не застосовувалися. Блокування fail2ban тепер поширюється і на TCP, і на UDP.
- **Автоматичне встановлення fail2ban при встановленні та оновленні панелі** — При встановленні та оновленні панелі на звичайному сервері fail2ban тепер встановлюється та налаштовується автоматично (раніше це відбувалося лише в Docker), щоб функція IP-ліміту працювала з коробки. Поведінкою керує змінна оточення XUI_ENABLE_FAIL2BAN: налаштування виконується, якщо змінна не задана або дорівнює true. Ручний запуск доступний командою x-ui setup-fail2ban; помилка fail2ban не перериває встановлення або оновлення.
- **Перевизначення порту панелі через змінну XUI_PORT** — Додано змінну оточення XUI_PORT, яка задає порт вебпанелі лише на час роботи поточного процесу, не змінюючи збережене значення webPort у базі. Допустимі значення від 1 до 65535; порожнє, некоректне або таке, що виходить за діапазон значення ігнорується (використовується webPort) з попередженням у лозі. При використанні Docker з bridge-мережею публікований порт контейнера має збігатися з XUI_PORT, наприклад XUI_PORT=8080 та ports: «8080:8080».
- **CLI: прапори -webCert/-webCertKey тепер застосовуються в підкоманді setting** — У CLI прапори -webCert та -webCertKey тепер працюють і в підкоманді x-ui setting (раніше вони мовчки ігнорувалися і панель залишалася без HTTPS). Вказавши їх, можна одразу задати шлях до сертифіката та ключа вебпанелі, не викликаючи окрему підкоманду cert.
- **Ім’я файлу резервної копії БД формується за адресою сервера** — Файли резервної копії бази даних тепер називаються за адресою сервера, а не фіксованими x-ui.db / x-ui.dump. При завантаженні з браузера ім’я береться з адреси панелі в адресному рядку, інакше — з налаштованого вебдомену, а за його відсутності — з публічного IP (спочатку IPv4, потім IPv6). Так бекапи з різних серверів легко розрізняти. Розширення залишається .db для SQLite та .dump для PostgreSQL.
- **Підтримка встановлення та оновлення на хостах лише з IPv6** — Скрипти встановлення та оновлення тепер коректно працюють на серверах лише з IPv6: завантаження релізу та допоміжних файлів більше не примусово використовує IPv4, тому панель можна встановити та оновити на хості без IPv4-адреси.
